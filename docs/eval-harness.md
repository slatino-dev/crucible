# Crucible eval-harness contract

> **This is a consumable contract, not a description.** Basilisk (project 5) implements security suites and scorers against this document with zero Crucible-internal context. The interface is Crucible's; the security scorers themselves are authored in Basilisk. Contract freezes at **v1.0** at Crucible v1 with an additive-only compatibility promise until v2; Basilisk sign-off is a Crucible release gate. Storage: D1 (drizzle-orm, versioned migrations). IDs: ULID. Times: UTC ISO-8601. All boundaries: Zod.

## Consumption mode (how Basilisk actually depends on this)
`harness/` ships as a **versioned package** (`@crucible/harness`, semver) that Basilisk installs and deploys **inside its own Worker with its own D1 / R2 / Workers AI bindings** — not a shared Crucible deployment, not a monorepo path import, not a live Crucible HTTP API. The package surface Basilisk imports and the bindings it must provide are the contract:
- **Exported code surface:** the `RunOrchestrator` DO class, the drizzle schema + migrations for every entity below, the scorer plugin interface (single-output families only — pairwise `Comparison` scoring is orchestrator-internal, not a plugin seam; see Scorer plugin interface), the pure stats module (CIs / paired test / rank-banding / FDR), and the Zod schemas for every boundary. Basilisk registers its own scorer implementations and suites; it changes none of the above.
- **Bindings Basilisk must declare:** a D1 database, an R2 bucket, a Workers AI binding, and a DO namespace bound to `RunOrchestrator`. The package is binding-agnostic (names injected via config), so Basilisk's own resources satisfy it.
- **Quota consequence (stated, not solved here):** if Basilisk deploys under the **same Cloudflare account** as Crucible, it draws the same shared daily Workers AI / D1 / KV pools (see ARCHITECTURE "Account-shared quota"). Split accounts or a budget-share allocation is the mitigation; the package itself imposes no cross-app coordination.
- **Semver promise:** the package follows the additive-only compatibility promise below; a Basilisk-breaking change is a major bump, never a silent minor.

## Data model

**Suite** `{id, slug, name, domain, cases_license, cases_provenance, judging, anchor_target_id?}` — logical grouping. `judging` ∈ `structural | single_rubric | pairwise`. **`anchor_target_id` is required when `judging = pairwise`**: pairwise scoring is *anchored*, not all-pairs — every candidate is compared against this one fixed reference target, so cost is O(M) in the model count, not O(M²) (see ARCHITECTURE "Neuron budget"). The anchor is itself a Target row; it appears on the board at a definitional 50% win-rate-vs-anchor (or is drawn as the zero line), never ranked against itself.
**SuiteVersion** `{id, suite_id, semver, content_hash, frozen_at}` — immutable once frozen; `content_hash` covers all member cases; a Run pins a SuiteVersion by hash. Editing a case means a new version, never a mutation.

**Case** `{id, suite_version_id, input, expected?, scorer_type, rubric_ref?, weight, tags[], param_template?, seed?}`
- `input`: typed JSON, one of `single_turn` (string/messages), `multi_turn` (message list), or `tool_use_trace` (declared per suite via a payload schema — **Basilisk's attack scenarios are a `tool_use_trace` payload shape, and fit here without contract change**).
- `expected`: optional golden (required by structural scorers, optional for judges with reference-anchored rubrics).
- `tags[]`: drive per-category rollups — **Basilisk sets OWASP Agentic Top 10 category tags here** to get per-category leaderboards for free.
- `param_template` + `seed`: for parametrized/contamination-resistant suites — each run instantiates fresh variants deterministically from the seed.

**Target** `{id, provider, model_id, params, version_label, fingerprint}` — `provider` is `workers-ai` in v1 (`http`/`byo-key` later, behind their security gates). `fingerprint` = hash of (model_id + params) for regression identity.
**Judge** `{id, model, prompt_template, prompt_version, protocol, k, aggregation, calibration_ref?}` — `protocol` ∈ `single_rubric | pairwise`. **Immutable once referenced by a Run.**

**Run** `{id, suite_version_hash, target_id, judge_config_hash?, anchor_run_id?, seed, sampling_params, n_trials, status, started_at, finished_at, neurons_spent, environment}` — the unit of execution; a full audit of exactly what ran. **`anchor_run_id`** (pairwise only) points at the anchor target's Run over the *same* `suite_version_hash`; a candidate's pairwise Run is judged against that pinned anchor Run, which is what lets two single-target Runs form one comparison.
**CaseResult** `{id, run_id, case_id, trial_idx, model_output_ref, latency_ms, tokens}` — `model_output_ref` points at R2. One row per case×trial.

**Score** `{id, case_result_id, scorer, verdict, judge_transcript_ref?, flags}` — the scoring unit for **structural** and **single_rubric** scorers (one target, one output).
- `verdict`: domain declared by scorer type — `pass|fail` (structural), numeric `scale` (rubric), or `severity` (security). **Declared, never free-form.** `severity` is an **ordered enum** `none < low < medium < high < critical`, mapped to fixed numeric levels `{none:0, low:1, medium:2, high:3, critical:4}` (the mapping is part of the contract, not the scorer's choice); its per-suite×target aggregate is **mean severity level**, and the paired-per-case regression test operates on per-case severity-level deltas exactly as it does on any numeric scale. Basilisk's author (Opus) reviews this severity declaration against real Basilisk code before freeze.
- `judge_transcript_ref`: R2 ref — **NOT NULL for any llm_judge score** (replayability is a schema constraint).
- `flags`: `self_judged`, `unscoreable`, etc.

**Comparison** `{id, case_id, judge_config_hash, case_result_candidate, case_result_anchor, verdict_order1, verdict_order2, verdict, transcript_ref_order1, transcript_ref_order2, flags}` — the scoring unit for **pairwise** judging. It records both operands (candidate vs anchor `CaseResult`s), both orders' verdicts, the swap-resolved `verdict` (a win only if the candidate is preferred in *both* orders, else `tie`; arXiv 2306.05685), and **both** transcript refs (NOT NULL — replayability). `flags` carries `self_judged`, `unscoreable`. A candidate Run and its `anchor_run_id` Run supply the two `CaseResult`s per case×trial.

**Aggregate** `{id, suite_version_id, target_id, judge_config_hash?, metric, value, ci_low, ci_high, n, provenance_ref}` — **computed only from Score or Comparison rows; `provenance_ref` is mandatory; never hand-settable.** The `metric` is fixed per scorer family so the leaderboard never mixes incomparable numbers: structural → **pass-rate**; single_rubric → **mean scale score**; pairwise → **win-rate-vs-anchor** (fraction of cases the candidate wins in both orders); severity → **mean severity level**. Every `value` carries a bootstrap 95% CI resampled over the **case** as the primary unit; when `n_trials > 1` the trials cluster within a case and the CI uses the clustered treatment of arXiv 2411.00640 (resample cases, then trials within case) so repeated trials never masquerade as independent evidence. `n` is the case count, not the row count.
**Baseline** `{id, suite_version_id, target_id, aggregate_ref, pinned_by, pinned_at}` — one pinned baseline per suite×target.
**Regression** `{id, suite_version_id, target_id, baseline_ref, run_id, p_value, q_value, effect_size, confirmation_run_id?, status, detected_at}` — an event row. `status` ∈ `unconfirmed | confirmed`. Because nightly re-runs repeat the same paired test against a fixed baseline (a growing family of tests), a raw single-night alpha hit is stored `unconfirmed` and is **published to the timeline as a *detected* regression only when `status = confirmed`** — i.e. it survives the **Benjamini–Hochberg FDR** control across the nightly family (`q_value` = the FDR-adjusted p-value), clears a minimum `effect_size`, AND is reproduced by the run at `confirmation_run_id`. This multiple-comparisons correction is what keeps a Type-I false positive from ever headlining as "the organically detected regression" (see PRD Methodology → repeated testing; ARCHITECTURE).

## Scorer plugin interface
Typed I/O, Zod schemas both directions:
```
scorer(input: CaseInput, output: ModelOutput, ctx: ScoreContext) => Promise<ScoreResult>
```
Built-in families: **structural** (schema-validity, exact-match, tool-call correctness — callcheck heritage; no LLM, injection-immune) and **llm_judge**. **Basilisk registers its own scorer implementations against this same signature** — the interface is Crucible's, the security scorers are Basilisk's. `ScoreResult` carries the verdict + optional transcript ref + flags; returning an unparseable/failed result yields an `unscoreable`-flagged Score, never a silent drop.

**Single-output only — pairwise is NOT a plugin seam (v1).** The signature above takes exactly one model `output` and returns one `ScoreResult`, so the plugin interface covers only the **single-output families**: `structural`, `single_rubric`, and `severity`. **Pairwise/`Comparison` scoring is produced by the `RunOrchestrator`, not through this interface** — the orchestrator holds both operands (the candidate and anchor `CaseResult`s over the shared `suite_version_hash`, joined via `anchor_run_id`), runs both orders, applies the swap-resolution rule, and writes `Comparison` rows itself. A consumer therefore **cannot register a custom pairwise scorer through this seam in v1**; exposing one would be an additive contract change (a distinct two-operand signature carrying the anchor output), not something the current interface implies. This is no constraint on the stated consumer: **Basilisk's security scorers are single-output `severity` scorers**, which register here unchanged.

## Judge protocol
- **single_rubric**: grade output against a rubric; k samples; aggregation (majority/mean) declared on the Judge. Produces `Score` rows.
- **pairwise-with-position-swap (anchored)**: every candidate is compared against the suite's single declared **anchor target**, never all-pairs — so a judged suite over M models costs O(M) comparisons, not O(M²) (this is what makes the free-tier neuron budget close; see ARCHITECTURE). Run the comparison in BOTH orders; a win is counted only when the candidate is preferred in both (arXiv 2306.05685); disagreement = tie; both transcripts stored. Produces `Comparison` rows; the leaderboard metric is **win-rate-vs-anchor** with a bootstrap CI over cases.
- **Verbosity/length bias** is mitigated as a first-class control, not left to the swap alone: the judge prompt forbids rewarding length, and calibration additionally reports a **length covariate check** (win-rate regressed on candidate−anchor output-length delta); a judge whose verdicts track length beyond a declared tolerance fails calibration and cannot headline. (Position bias → swap; self-preference → family rule below; verbosity → this control. All three named biases from arXiv 2306.05685 now have a mitigation.)
- Judge output is parsed through a small Zod schema (verdict enum + rationale) with bounded retry, then `unscoreable`. Untrusted case/model content is passed inside labeled data regions the system prompt marks as non-instructions (injection hardening — **see ARCHITECTURE Security posture, the Opus-owned authority for this control; this line is a pointer, not a second source of truth**).
- **Judge/target family rule:** a judge never headline-scores a target of its own model family (self-preference; arXiv 2404.13076). The allowed judge→target matrix is declared per suite (e.g. a Qwen or Gemma judge for Llama-family targets and vice-versa); a suite with no in-catalog cross-family judge that clears calibration cannot headline a judged metric and says so honestly.
- Transcript storage is **mandatory** for every judge score.

## Regression semantics (eval-gate absorbed)
Two related-but-distinct comparisons, both the same paired-per-case test (arXiv 2411.00640) at a declared alpha + minimum effect size, deliberately **not** conflated in the vocabulary:
- **Regression (same identity, over time).** Baseline pinned per `suite_version × target`, where a target's identity is `fingerprint = hash(model_id + params)`. A later Run of the **same** `suite_version × target` that drops significantly is a **regression** → `Regression` event row + timeline entry. This is the only comparison the word "regression" names.
- **Multiple-comparisons control (the nightly stream is a family of tests).** Nightly re-runs repeat the same paired test against a fixed baseline, so raw per-night alpha controls neither the family-wise nor the false-discovery rate; a `Regression` publishes as a *detected* regression (`status = confirmed`) only after **Benjamini–Hochberg FDR** adjustment across the rolling nightly family, a minimum effect size, and a **confirmation re-run** reproduce the drop. The `q_value`, `effect_size`, and `confirmation_run_id` are persisted on the Regression row and shown on the timeline; an unconfirmed hit is rendered as **unconfirmed**, never as a regression. **Basilisk inherits this unchanged** — a security-severity regression is subject to the same correction, so a single noisy night never headlines "the agent got more vulnerable."
- **A/B variant diff (two declared identities, same case set).** Comparing two *different* identities — a base vs a quantized variant (different `model_id`), or an intact vs a degraded prompt format (different `params`/`suite_version`) — over the same cases is a **variant diff**, not a regression, because there is no shared baseline identity. It runs the identical paired test across the two Runs and is labeled a **comparison** in data and UI. This is the mode the v0.5 induced demo uses (and it is disclosed as *induced*, per the honesty brand).
- Both surface a **gate verdict as a first-class API object** (`{status: pass|regression|diff, kind: regression|variant_diff, p_value, effect_size, baseline_ref?, run_ref, compared_run_ref?}`), returned 200 (a verdict, not an error), and reflected by the SVG badge. **The same semantics serve Basilisk as "did the agent get more vulnerable"** — a severity-metric regression is a security regression with no contract change; a base-vs-hardened variant diff is a security A/B with none either.

## Basilisk-consumption checklist (v1.0 sign-off)
These are **claims to be verified against real Basilisk code, not pre-granted checkmarks** — they stay unchecked (☐) until Opus signs off, because a checked box before the consumer exists is exactly the fabricated-status failure this project's charter forbids.

3. ☐ A security scorer (severity verdict, per the ordered-enum declaration above) registers against the scorer interface unchanged.
4. ☐ The paired-test regression gate fires on a severity-metric drop.
5. ☐ Run orchestration (the `RunOrchestrator` DO, budget check, transcript storage, provenance) is consumed by importing `@crucible/harness` into Basilisk's own Worker with Basilisk's own bindings (Consumption mode above) — no fork, no edit.

## Compatibility promise
Additive-only within v1.x: new optional fields, new scorer families, new verdict domains — never a rename, never a required-field addition, never a semantics change to an existing field. Breaking changes = contract v2 + a migration note. Suite versions and judge configs are immutable by construction, so historical runs remain replayable forever.
