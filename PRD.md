# PRD: Crucible — public LLM evals arena with honest leaderboards

> Status: approved charter (2026-07-08). One-liner: **the leaderboard that shows its error bars.** Name alternates considered: Assay (precise metaphor, weak as a spoken brand), Gauntlet (collides with security tooling), Proving Ground (flabby). Crucible wins on metaphor (the vessel that assays metal under heat), one-word strength, and continuity with the absorbed v1 stub. Absorbs: Crucible stub (arena concept), eval-gate (regression-gate semantics → gate API + status badges), callcheck (tool-call/schema correctness → structural scorer family, no LLM judge).

## Problem
Public LLM leaderboards are broadly distrusted, and the distrust is now documented: the most-cited arena is distorted by selective score disclosure, undisclosed private-variant testing, and data-access asymmetries (see Evidence). The flagship open leaderboard (Hugging Face Open LLM Leaderboard) was retired ("completed, retired, dead") in a March 2025 maintainer post (huggingface.co/spaces/open-llm-leaderboard/open_llm_leaderboard/discussions/1135, seen 2026-07-08). Live open leaderboards do exist — **LiveBench** (arXiv 2406.19314; live at livebench.ai; monthly-refreshed, contamination-limited, objective ground-truth scoring, no LLM judge) is the strongest — but they solve *contamination*, not *uncertainty*: none publishes confidence intervals on scores, runs a documented LLM-judge-bias program, or exposes click-through transcript provenance and regression gates. Meanwhile teams' internal evals are ad-hoc: single-run point scores with no variance, LLM judges with unmitigated position/verbosity/self-preference biases, no regression tracking across model or prompt versions. Nobody publicly combines suite-based measurement + variance reporting (error bars) + bias-mitigated LLM-as-judge + full transcript replayability + regression gates — on infrastructure anyone can afford. That combination is this product.

## Evidence (verified 2026-07-08; sources re-confirmed this session)
- **"The Leaderboard Illusion"** (arXiv 2504.20879, May 2025; NeurIPS 2025) — Chatbot Arena distortions: 27 private Meta LLM variants tested pre-Llama-4 with selective disclosure; Google and OpenAI received an estimated 19.2% and 20.4% of all arena data respectively while a combined 83 open-weight models got an estimated 29.7%. Re-verified via arxiv.org/abs/2504.20879 this session.
- **"Adding Error Bars to Evals"** (Anthropic, arXiv 2411.00640, Nov 2024) — the statistical spine: standard errors and CIs on eval scores, clustered SEs when trials cluster by question, paired-difference tests for model comparison, power analysis for suite sizing. Crucible operationalizes this paper in public.
- **"Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena"** (Zheng et al., arXiv 2306.05685, NeurIPS 2023) — names position, verbosity, self-enhancement bias; prescribes position-swap ("only declare a win when preferred in both orders"). Crucible's pairwise protocol implements exactly this.
- **"LLM Evaluators Recognize and Favor Their Own Generations"** (Panickssery et al., arXiv 2404.13076, NeurIPS 2024) — self-preference correlates with self-recognition; grounds the rule that a judge never headline-scores its own model family.
- **promptfoo** (github.com/promptfoo/promptfoo, MIT) — proves eval-harness demand and is the fork-first candidate (see oss-decisions). Star count NOT re-verified this session — cited as "widely adopted," not a number.
- *Cut for lack of a re-verified source this session:* the exact HF-leaderboard "~13K models over 2 years" figure and the ArenaHard "up to 112% relative gains" figure — referenced qualitatively only.

## Methodology (the honest-leaderboard spec — this section *is* the /methodology page)
The charter sells measurement honesty, so the method is a first-class contract, not scattered asides. Every rule here is implemented by the stats module and rendered on /methodology with footnotes to the papers.
- **Metric per scorer family (never mixed on one axis):** structural → **pass-rate**; single_rubric → **mean scale score**; pairwise → **win-rate-vs-anchor** (fraction of cases the candidate beats the suite's one fixed anchor target in *both* orders); severity (Basilisk) → **mean severity level** on the ordered enum `none<low<medium<high<critical` (0–4). Definitions live in eval-harness.md so Basilisk inherits them.
- **Uncertainty:** every headline number carries a **bootstrap 95% CI resampled over cases** as the primary unit. When `n_trials > 1`, trials cluster within a case, so the CI uses the **clustered treatment** of arXiv 2411.00640 (resample cases, then trials within case) — repeated trials never inflate apparent precision. `n` reported is the case count.
- **Rank-band criterion:** a strict rank between two models is shown **only when their 95% CIs do not overlap**; overlapping models render as a **tie-band** at the same rank. The board's insight title says so ("ranks shown only where intervals separate").
- **Pairwise aggregation decision (settled):** **anchored, not all-pairs.** Each candidate is judged against one fixed anchor target → O(M) comparisons, aggregating cleanly to win-rate-vs-anchor with a per-case bootstrap CI. All-pairs O(M²) was rejected on the neuron budget (see ARCHITECTURE). No ELO/Bradley-Terry: those infer a latent-skill scalar from sparse comparisons and hide uncertainty behind a point rating — the opposite of this product's thesis.
- **Judge-bias program (all three named biases from arXiv 2306.05685 have a mitigation):** *position* → mandatory order-swap, win only if preferred both ways; *self-preference* (arXiv 2404.13076) → a judge never headline-scores its own model family, enforced by the declared judge→target family matrix; *verbosity/length* → the judge prompt forbids rewarding length **and** calibration runs a length-covariate check (win-rate regressed on candidate−anchor output-length delta); a judge whose verdicts track length beyond tolerance fails calibration.
- **Calibration gating rule:** every judge config is scored against the human golden set (sized by the arXiv 2411.00640 power analysis); Cohen's kappa vs human — **and** inter-annotator kappa from a second-annotator subsample — are published; **a judge below the agreement threshold, or failing the length-covariate check, cannot headline** and is shown as "below calibration threshold." The judges have a leaderboard too.
- **Repeated/nightly testing (multiple-comparisons control) — the correction that makes an "organically detected regression" defensible.** Nightly paired tests against a fixed baseline per suite×target form a *growing family* of tests, so raw per-night alpha does not control the family-wise or false-discovery rate: at alpha=0.05 a ~90-night window per series yields ~4–5 expected **false-positive** "significant regressions" by chance. For a product whose thesis is statistical honesty, a raw single-night alpha hit is therefore **not** a published regression. Two controls compose: (1) the nightly stream is treated as a family and adjusted by **Benjamini–Hochberg FDR** over a rolling window, with the target false-discovery rate stated on /methodology; and (2) a surviving candidate must additionally clear a **minimum effect size** AND be **reproduced by a confirmation re-run** before it publishes as a *detected* regression. The raw p-value, the FDR-adjusted q-value, the effect size, and the confirmation run are stored on the Regression entity (eval-harness.md) and rendered on the timeline; a hit that fails correction or confirmation is shown as **unconfirmed**, never as a regression. This is what lets the organic-regression proof above be a genuine signal rather than a self-disproving Type-I error, and the correction itself is footnoted on /methodology.

## v0.5 cut (smallest version that tests the hypothesis) — target 2026-09-08
Hypothesis: *a statistically honest, fully replayable LLM-judge eval pipeline can run end-to-end on Cloudflare's free tier, and its paired-per-case test detects a real difference.* The smallest test of *that* hypothesis is one judge, one judged suite, a few targets, and the stats — nothing about the judge loop is tested by a second (structural) suite or by 6 targets, and the interactive board is launch surface, not hypothesis coverage. So v0.5 is deliberately narrow: one Worker + D1 + one RunOrchestrator DO + the BudgetLedger DO.
- **Stats module FIRST**, pure functions with golden tests vs hand-computed values: bootstrap 95% CIs, paired-per-case comparison (arXiv 2411.00640), rank-banding, clustered-trial treatment. Property tests for invariants (CI narrows monotonically with n; paired test symmetric).
- **One judged suite** (instruction-following), **anchored pairwise** with position swap + k=3, **3 Workers AI targets** (1 is the anchor), n_trials ≥3. Runs via CLI/API only — no public button.
- **The paired gate proven via an A/B variant diff, labeled induced.** Take one target and run its cases through a deliberately **degraded prompt format** (a *different declared identity*, so this is a **variant diff**, not a same-identity regression — see eval-harness Regression semantics), then show the paired test detects the drop at the declared alpha + effect size. The changelog states plainly that this difference was **induced to demonstrate the gate**; the word "regression" is reserved for organic same-identity drift caught later by nightly runs (co-primary B).
- **One honest chart** (a static SVG render with CI whiskers is sufficient evidence at v0.5): 3 targets ranked with error bars, the induced variant diff shown, every score clickable to its stored transcript.

Moved out of v0.5 to keep the 2026-09-08 date safe: the **structural tool-call suite** (callcheck heritage — cheap, but tests nothing in the judge hypothesis) → v0.75; the **interactive tie-band/click-through leaderboard** → v1.

## Existential risk: judge viability (named, with a fallback)
The judged product loop dies if **no free Workers AI judge config clears the calibration threshold** against the human golden set, and the risk is compounded by two things: the self-preference rule forbids a judge headline-scoring its own family, yet the Workers AI catalog is Llama-heavy; and the allowed judge→target family matrix is real work, not a footnote. Mitigation, decided now: (1) **structural suites always headline** regardless of judge outcome — tool-call correctness, schema validity, and objective ground-truth cases need no judge and are injection-immune, so the board is never empty; (2) a judged metric that cannot find an in-catalog cross-family judge clearing calibration **displays "below calibration threshold" honestly** rather than headlining a biased number — which is itself the on-brand outcome (the product that shows its error bars also shows when it *can't* trust its judge); (3) the **judge→target family matrix is drafted before committing to 8 models** (e.g. Qwen or Gemma judges for Llama-family targets and vice-versa; the matrix lives in eval-harness.md and gates which judge may headline which targets). Verified this session that the honest-fallback path is viable because objective-scored suites (LiveBench's approach) need no judge at all.

## Activation event
**`live_run_completed`** — a visitor-initiated spotlight eval run reaches scored completion, result + CI rendered in-session. It is the exact portfolio north-star shape (recruiter/founder session reaching a live demo interaction), genuinely measurable (D1 run record joined to the PostHog event, no vanity pageview proxy), and forces the right decisions (suite small enough to finish **<40s p95** on the allocated neuron share, streaming progress worth watching). TTV target: **<60 seconds** from landing to a completed scored run (a <40s run leaves ~20s for landing, reading, and clicking — the two numbers are now mutually consistent). Typed events.ts funnel — ordered by the path a visitor actually takes (run first, audit after): `arena_viewed → live_run_started → live_run_completed → repo_clicked`. **`score_provenance_opened`** is tracked as a **parallel engagement event, not a strict funnel step**, because most visitors run before they audit and a strict ordering that puts provenance before the run would undercount activation.
